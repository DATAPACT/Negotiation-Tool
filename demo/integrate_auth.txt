Important design
- authentication-service is the single auth source
- negotiation-api and contract-service do not create their own independent auth flow anymore
- the same access_token from login is reused across services

Important code idea
- Keep one central authentication issuer:
  - authentication-service issues and verifies tokens
- Other services should not trust their own SECRET_KEY for user login tokens:
  - they must call authentication-service /user/verify-token/
- Keep local proxy auth endpoints when useful for Swagger/docs:
  - negotiation-api /user/login/ -> forwards to authentication-service
  - negotiation-api /user/register -> forwards to authentication-service
  - contract-service /user/login/ -> forwards to authentication-service
  - contract-service /user/register -> forwards to authentication-service
- Protected business endpoints use bearer token dependency:
  - Django stores access_token in session
  - browser/Django sends Authorization: Bearer <token>
  - negotiation-api / contract-service verify the token with authentication-service
- Service-to-service calls must forward the same user token:
  - Negotiation-Tool -> contract-service
  - negotiation-api -> contract-service
- Contract authorization needs two layers:
  - authentication: token is valid
  - authorization: current user is allowed to access this contract
- To support authorization in contract-service, contract documents now keep:
  - owner_user_id
  - owner_username_email
  - authorized_user_ids
- When contract data is created or updated, preserve access metadata instead of overwriting it
- When remote external dashboard integrations fail:
  - first separate local auth problems from remote service errors
  - if token call is 200 and next API is 500, the remaining issue is remote payload/server-side, not local authentication
