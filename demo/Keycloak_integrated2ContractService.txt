Keycloak Integrated to Contract Service
======================================

1. Purpose of the Keycloak code
The Keycloak-related code added into `Contract_Service` is designed for a global SSO scenario.
It allows an external platform or parent portal to send a bearer token to `contract-service`,
and then `contract-service` checks that token with Keycloak.

Relevant file:
- `../Contract_Service/auth.py`

Main idea in that file:
- read `KEYCLOAK_ISSUER`
- receive `Authorization: Bearer <token>`
- call:
  `<KEYCLOAK_ISSUER>/protocol/openid-connect/userinfo`
- if Keycloak accepts the token, allow the request


2. What "global Keycloak" means
Global Keycloak means:
- one central identity provider exists outside your service
- many systems trust that same Keycloak server
- users sign in once
- all connected systems can reuse the same SSO token

So Keycloak is the enterprise/global identity authority.


3. What `authentication-service` means in your current project
Your `authentication-service` is now the local/project authentication authority for:
- `Negotiation-Tool`
- `negotiation-api`
- `contract-service`

It currently:
- handles login/register/logout/details
- issues the JWT used by your services
- verifies that JWT through `/user/verify-token/`

So in the current working architecture, `authentication-service` is the token issuer that your own services trust.


4. Relationship between `authentication-service` and Keycloak
They are both authentication authorities, but they are not automatically the same thing.

Relationship:
- Keycloak can be an upstream/global SSO provider
- `authentication-service` can be your local/project auth gateway

There are two possible architectures:

A. Independent local auth architecture
- `authentication-service` is the main login/token service
- other services trust `authentication-service`
- Keycloak is not required

B. Global SSO architecture
- Keycloak is the main login/token service
- your `authentication-service` may become:
  - a proxy to Keycloak, or
  - a token translator / user-sync layer, or
  - unnecessary if all services trust Keycloak directly

Important:
- if `authentication-service` issues one kind of token
- and Keycloak issues another kind of token
- then a service must know which issuer it is supposed to trust

That is why mixing both models without a clear design causes conflicts.


5. Current working model in your codebase
Your current code is working with this model:

`Negotiation-Tool` -> login to `authentication-service`
`negotiation-api` -> verify token with `authentication-service`
`contract-service` -> verify token with `authentication-service`

In this model, the old global Keycloak dependency should not be applied globally to all routes.


6. Why the old Keycloak dependency was risky in `Contract_Service`
Previously, the partner code was intended to protect the whole FastAPI app by adding:

`dependencies=[Depends(verify_access)]`

That means:
- every request may be checked by Keycloak logic first
- but your current token comes from `authentication-service`
- so a valid `authentication-service` token may fail in Keycloak validation

This is why the Keycloak code and the new `authentication-service` code can conflict if both are used as primary auth validators.


7. How to understand the Swagger `/docs` iframe code
The custom `/docs` page in `Contract_Service` was designed for embedded SSO usage.

Idea:
- parent portal opens Swagger `/docs` in an iframe
- parent portal already has a Keycloak token
- parent portal sends token via `window.postMessage`
- Swagger adds `Authorization: Bearer <token>` automatically
- backend validates it

This is useful when:
- your API docs are opened inside another platform
- the user should not type credentials manually again

10. How `authentication-service` and Keycloak can work together
If you want both in one final system, you need a clear ownership rule.

Recommended options:

Option 1: `authentication-service` is primary
- users log in through `authentication-service`
- all internal services trust `authentication-service`
- Keycloak is not used for internal token verification
- easiest for your current codebase

Option 2: Keycloak is primary
- users log in through Keycloak
- `authentication-service` becomes a wrapper/proxy around Keycloak
- `authentication-service` may create local user records and profile mapping
- all services either:
  - trust Keycloak directly, or
  - trust only `authentication-service` if it validates Keycloak tokens centrally

Best rule:
- only one system should be the primary token validator for service-to-service auth


11. Practical workflow if you keep the current project design

Workflow:
1. User logs in through `authentication-service`
2. `authentication-service` returns `access_token`
3. `Negotiation-Tool` stores that token in session
4. When calling `negotiation-api` or `contract-service`, send:
   `Authorization: Bearer <access_token>`
5. Those services call:
   `API_AUTHENTICATION_URL/user/verify-token/`
6. If valid, request continues
7. If invalid, return `401`


12. Practical workflow if you want future Keycloak integration

Better workflow:
1. User logs in with Keycloak
2. Keycloak returns SSO token
3. `authentication-service` receives that token
4. `authentication-service` validates it with Keycloak
5. `authentication-service` maps or creates the local user
6. `authentication-service` returns either:
   - the same Keycloak token, or
   - a local project token tied to the Keycloak identity
7. Other services trust one chosen token source only

This way Keycloak becomes the upstream identity provider, and `authentication-service` becomes the project integration layer.


13. Short summary
- Keycloak code in `Contract_Service` was written for enterprise/global SSO
- your current working project uses `authentication-service` as the main auth authority
- both can coexist only if their responsibilities are clearly defined
- for your current code, the cleanest solution is:
  `Negotiation-Tool`, `negotiation-api`, and `contract-service` all trust `authentication-service`
- later, if needed, Keycloak can be connected above `authentication-service` as the global login provider
